Friday

An autonomous agent that controls macOS through a perception-action loop: screen capture, UI understanding, and synthetic input generation.

2025 · Solo project · macOS
Problem

LLMs can reason but can't act

Language models produce text. But most real-world tasks require interacting with graphical interfaces: clicking buttons, filling forms, navigating menus, chaining actions across applications. The gap between language understanding and physical interface control is the bottleneck.

Approach

Perceive, plan, execute, verify

Friday implements a closed-loop agent architecture. Each cycle: capture the screen, extract UI state, plan the next action via an LLM, execute it via synthetic input, then verify the result before proceeding.

1. Screen Capture → raw pixels at configurable FPS
2. Vision Module → OCR + element detection + layout parsing
3. AX Bridge → macOS Accessibility API tree traversal
4. State Fusion → merge visual + accessibility into structured UI state
5. LLM Planner → task decomposition + action selection
6. Executor → synthetic mouse/keyboard events via CGEvent
7. Verifier → re-capture, diff, confirm action succeeded
↓ loop

The key design choice: dual perception. Vision (OCR) handles what's visible on screen. The Accessibility API provides the structural UI tree (button labels, text fields, hierarchies) that vision can miss. Fusing both gives more robust state understanding than either alone.

Implementation

Technical details

  • Screen capture via CGWindowListCreateImage (low-level, no overhead from screencapture CLI)
  • OCR: Tesseract for text extraction, custom heuristics for element boundary detection
  • Accessibility: AXUIElement API via pyobjc for UI tree traversal (roles, labels, positions, states)
  • LLM backend: swappable between GPT-4, Claude, local models. Structured output for action primitives
  • Input synthesis: CGEventCreateMouseEvent / CGEventCreateKeyboardEvent for precise coordinate-level control
  • Error recovery: state comparison before/after each action, retry with alternative strategy on failure
Connection to 20n

From screens to bodies

Friday's architecture maps directly to embodied robot control. The perception-action loop is the same structure:

  • Friday: screen pixels → UI understanding → plan → mouse/keyboard
  • Robot: camera frames → scene understanding → plan → actuators

Building Friday was the project that made me think seriously about the perception-action loop as a general computational pattern — which led directly to exploring Active Inference for robot control.
